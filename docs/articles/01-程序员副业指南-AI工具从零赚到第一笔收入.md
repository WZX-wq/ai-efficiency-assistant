## 第一步：市场分析——找到你的"小而美"赛道

### 为什么选择 AI 效率工具？

做副业最忌讳的就是"什么火做什么"。我选择 AI 效率工具赛道，基于以下几个判断：

1. **需求真实存在**：每天有上亿人在浏览器中处理文字——写邮件、写文档、读论文、翻译资料。这些场景天然需要 AI 辅助。
2. **技术门槛适中**：调用 AI API 不需要训练模型，一个前端工程师就能独立完成。
3. **竞争格局未定**：虽然 ChatGPT、Claude 等大厂产品很强，但它们是通用工具。垂直场景下的轻量级工具依然有巨大的生存空间。
4. **边际成本低**：SaaS 产品的边际成本趋近于零，API 调用费用极低。

### 竞品分析

我调研了市面上主流的 AI 写作助手：

| 产品 | 形态 | 优势 | 不足 |
|------|------|------|------|
| Grammarly | 浏览器扩展 | 英文语法强大 | 不支持中文，价格高 |
| Monica | 浏览器扩展 | 功能丰富 | 免费额度少，加载慢 |
| 沉浸式翻译 | 浏览器扩展 | 翻译体验好 | 功能单一 |
| ChatGPT | Web 应用 | 能力最强 | 需要切换窗口，打断工作流 |

**我的切入点**：做一个轻量级的 Chrome 扩展，选中文字后一键调用 AI 进行改写、扩写、翻译、总结。不切换窗口，不打断工作流。

### AI 接入：白山智算 + DeepSeek-V3

为什么选择白山智算而不是直接用 OpenAI？

1. **价格优势**：白山智算的 DeepSeek-V3 价格远低于 GPT-4，适合个人开发者
2. **兼容 OpenAI API**：接口格式完全兼容，迁移成本为零
3. **中文能力强**：DeepSeek-V3 在中文场景下表现优秀
4. **国内访问稳定**：不需要翻墙，API 响应速度快

```typescript
// API 调用核心代码
const DEFAULT_API_URL = 'https://api.edgefn.net/v1/chat/completions';

export async function callDeepSeekAPI(
  prompt: string,
  settings: ExtensionSettings,
): Promise<string> {
  const { apiKey, model, temperature } = settings;

  const requestBody: DeepSeekRequest = {
    model: model || 'DeepSeek-V3',
    messages: [
      {
        role: 'system',
        content: '你是一个专业的文字处理助手。请根据用户的要求对文字进行处理，直接输出结果，不要添加多余的解释。',
      },
      {
        role: 'user',
        content: prompt,
      },
    ],
    temperature: temperature ?? 0.7,
    max_tokens: 2048,
    stream: false,
  };

  const response = await fetch(apiUrl, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      Authorization: `Bearer ${apiKey}`,
    },
    body: JSON.stringify(requestBody),
  });

  const data: DeepSeekResponse = await response.json();
  return data.choices[0].message.content.trim();
}
```

### 后端：Node.js + Express

虽然 Chrome 扩展可以直接调用 API，但我还是搭建了一个简单的后端服务：

```typescript
// server/src/index.ts
import express from 'express';
import cors from 'cors';
import { rateLimit } from './middleware/rateLimit';
import { authMiddleware } from './middleware/auth';

const app = express();

app.use(cors({ origin: '*' }));
app.use(express.json({ limit: '1mb' }));

// 速率限制：每分钟最多 20 次请求
app.use('/api/ai', rateLimit(20, 60 * 1000));
app.use('/api/ai', authMiddleware);
app.use('/api/ai', aiRoutes);

app.listen(PORT, () => {
  console.log(`服务已启动: http://localhost:${PORT}`);
});
```

后端的作用：
- **API Key 安全**：避免在前端暴露 API Key
- **速率限制**：防止滥用
- **请求转发**：解决跨域问题

### Chrome 扩展架构

Chrome 扩展采用 Manifest V3 架构，包含三个核心部分：

```
ai-efficiency-assistant/
├── src/
│   ├── background/index.ts    # Service Worker（后台服务）
│   ├── content/index.tsx       # Content Script（注入网页）
│   ├── popup/Popup.tsx         # Popup（弹窗界面）
│   ├── components/             # 公共组件
│   ├── types/                  # TypeScript 类型定义
│   ├── utils/                  # 工具函数
│   └── manifest.ts             # 扩展配置
├── web/                        # Web 版本
├── server/                     # 后端服务
└── vite.config.ts              # Vite 构建配置
```

**关键设计决策**：

1. **Content Script 负责交互**：在用户选中文本时显示浮动按钮和操作面板
2. **Background Service Worker 负责 API 调用**：集中管理 API 请求，避免 Content Script 被页面 CSP 策略拦截
3. **Popup 作为完整操作界面**：提供更丰富的设置和操作选项

## 第四步：部署上线——零成本方案

### Chrome 扩展发布

Chrome 扩展的发布流程：

1. **构建产物**：`npm run build` 生成 `dist/` 目录
2. **打包**：将 `dist/` 目录打包为 `.zip` 文件
3. **上传**：在 [Chrome Web Store Developer Dashboard](https://chrome.google.com/webstore/devconsole) 上传

### Web 版本部署（Vercel 免费版）

```bash
# 配置 vercel.json 支持 SPA 路由
```

## 踩坑记录

### 坑 1：Chrome 扩展的 CSP 限制

Content Script 注入到网页中，会受到页面 CSP（Content Security Policy）的限制。解决方案是将 API 调用放在 Background Service Worker 中，Content Script 通过 `chrome.runtime.sendMessage` 发送请求。

```typescript
// Content Script 发送消息
chrome.runtime.sendMessage({
  type: 'AI_ACTION',
  action: actionType,
  selectedText,
});

// Background Service Worker 处理请求
chrome.runtime.onMessage.addListener((message, _sender, sendResponse) => {
  if (message.type === 'AI_ACTION') {
    handleAIAction(message.action, message.selectedText)
      .then(result => sendResponse({ type: 'ACTION_RESULT', result }))
      .catch(error => sendResponse({ type: 'ACTION_RESULT', error: error.message }));
    return true; // 保持异步通道
  }
});
```

### 坑 2：Manifest V3 的 Service Worker 生命周期

Manifest V3 的 Service Worker 不是常驻的，会在空闲时被终止。这意味着你不能在全局作用域中保持状态。所有状态都需要存储在 `chrome.storage` 中。

### 坑 3：API Key 安全

绝对不要将 API Key 硬编码在前端代码中。我的方案是：
- Chrome 扩展：用户自己配置 API Key，存储在 `chrome.storage.local` 中
- Web 版本：通过后端代理转发请求，API Key 存储在服务端环境变量中

## 给程序员的副业建议

1. **不要等准备好了再开始**：MVP 不需要完美，能跑就行
2. **选择你熟悉的技术栈**：副业不是学新技术的时候，效率第一
3. **先做产品，再做内容**：有产品才有内容，有内容才有流量
4. **重视部署和上线**：能在线上访问的产品才有价值
5. **持续迭代**：第一版不需要功能完善，但要快速上线获取反馈
